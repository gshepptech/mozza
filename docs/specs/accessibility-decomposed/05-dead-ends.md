# Domain 5: Dead Ends & Missing Feedback

**Wave:** 4 (parallel with Domains 4 and 6)
**Gaps covered:** D1–D10 (10 dead ends and silent failures)
**Depends on:** Domain 8 (shared components)

## Purpose

No screen should leave the user wondering "what happened?" or "what do I do now?"
Every action should have visible feedback. Every empty state should have a next step.

## Requirements

### FR-5.1: Alias Creation Error Display (D1)

**Where:** OrderWizard.tsx:250-270

**Current:** `catch {}` — alias creation fails silently.

**Change:**
- Add error state: `const [createError, setCreateError] = useState("")`
- In catch: `setCreateError("Couldn't add that app. Check the name and URL and try again.")`
- Display error below the form in red text
- Clear error when user modifies inputs
- If error contains "duplicate" or "already exists": "An app with that name already exists. Try a different name."

**Files:** OrderWizard.tsx:250-270

### FR-5.2: Empty Menu Smart Redirect (D2)

**Where:** OrderWizard.tsx:389-402

### FR-5.3: GitHub Scan "No Results" Guidance (D3)

**Where:** GitHubImport.tsx

**Current:** Shows warnings but no "what to do next" when scan finds nothing deployable.

**Change:**
```
We didn't find deployment files in this repo.

This usually means your app hasn't been set up for deployment yet.

What to do:
  • If you have a Docker image already → use the "Your menu" tab instead
  • If your developer has a Dockerfile → ask them to push it to the repo
  • If you're just testing → try one of our classic templates instead

[Switch to Classics →]
```

**Files:** GitHubImport.tsx

### FR-5.4: GitHub "Needs Build" Actionable Guidance (D4)

**Where:** GitHubImport.tsx

**Current:** Shows build instructions (shell commands) that a non-engineer can't execute.

**Change:**
- Replace shell commands with plain-English next steps:
  ```
  This app needs to be built into a package first.

  Ask your developer to:
    1. Build the app package (Docker image)
    2. Push it to a package registry (like GitHub Packages or Docker Hub)
    3. Give you the package URL (looks like: ghcr.io/yourorg/app:v1.0)

  Then come back and paste the URL in the "Your menu" tab.
  ```
- Keep technical build instructions in expandable "For developers" section

**Files:** GitHubImport.tsx:248-259

### FR-5.5: Deploy Failure Detail (D5)

**Where:** DeployWizard.tsx:40-47

**Current:** Generic messages like "Recipe has syntax errors — check your recipe and try again."

**Change:**
- For recipe parse errors: highlight the specific line/section that failed
  - Parse the error response for line numbers
  - Show: "There's a problem on line 5 of your recipe: unexpected 'xyz'"
- For plan build errors: "Your recipe has a configuration problem: {specific issue}"
- For plan validation errors: "Something doesn't look right: {specific issue}"
- For all errors: add "Edit Order" button that takes user back to Step 2

**Files:** DeployWizard.tsx:40-47

### FR-5.8: Auto-Scaling Impact Feedback (D8)

**Where:** StepAnythingElse.tsx

**Change:**

**Files:** StepAnythingElse.tsx:256-321

### FR-5.9: Database Engine Switch Warning (D9)

**Where:** StepDependencies.tsx

**Current:** User can switch engines freely with no warning.

**Change:**
- On first deploy (no existing deployment): no warning needed, just switch
- On re-deploy / edit (if deployment exists): show warning:
  "Switching database engines may cause data loss. Your existing data won't transfer automatically."
- Confirmation required to switch: "I understand, switch anyway"

**Files:** StepDependencies.tsx:130-154

### FR-5.10: GitHub Token Guidance (D10)

**Where:** GitHubImport.tsx

**Current:** Token input shown with "Never stored server-side" but unclear when needed.

**Change:**
- Hide token input by default
- Show only when scan fails with auth error: "This looks like a private repo. Paste a GitHub token to continue."
- Add "How to create a token" expandable:
  1. Go to GitHub → Settings → Developer settings → Personal access tokens
  2. Generate new token with "repo" scope
  3. Copy and paste here
- Add link: "Create a token on GitHub →"

**Files:** GitHubImport.tsx:102-126

## Acceptance Criteria

- [ ] Alias creation shows error message on failure (not silent)
- [ ] GitHub scan with no results shows clear next steps with tab links
- [ ] GitHub "needs build" shows non-engineer instructions, not shell commands
- [ ] Deploy failures show specific error details, not generic messages
- [ ] Recipe validation errors reference specific lines/sections
- [ ] DATABASE_URL generation is visibly announced
- [ ] Database engine switch warns about data loss (on re-deploy only)
- [ ] GitHub token is hidden until needed

## Observable Truths

- OT-1: A user whose alias creation failed sees a red error message explaining why
- OT-2: A user whose GitHub scan found nothing has 3 clear options to continue
- OT-3: A user who can't build an image knows to ask their developer for the URL
- OT-4: A user editing a recipe sees exactly which line has the error
- OT-5: A user who enabled a database understands why DATABASE_URL appeared
- OT-6: A GitHub token field only appears when the user actually needs one
