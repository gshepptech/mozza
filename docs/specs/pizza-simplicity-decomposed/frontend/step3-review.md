## Purpose

Build Step 3 of the fast lane: "Review your order" — a receipt-style order summary with collapsed recipe view and prominent "Place Order" button.

## Scope

**Included:**
- Receipt-style order summary card
- Collapsed "View recipe" section with syntax-highlighted .mozza source
- "Place Order" button with pizza styling
- Back button to return to Step 2
- Recipe generation from wizard state (calls existing `generateOrderRecipe`)

**Excluded:**
- Deploy execution (handled by DeployWizard.tsx's onComplete callback)
- Order tracker / deploy progress (handled by order-tracker domain)
- Recipe validation API calls (keep existing inline validation if present)

## Requirements

- REQ-1: Step 3 title: **"Review your order"**
- REQ-2: **Order summary card** styled as a receipt/ticket:
  - Header: app name in bold, combo type as a badge (e.g., "Website", "API + Database")
  - Line items section:
    - "Deploying to: [Local / Kitchen name]"
    - "Type: [combo label]" (e.g., "Website — a site people visit")
    - For each enabled dependency: "+ PostgreSQL database", "+ Redis cache"
    - "Public: Yes/No" (only if Kitchen target)
  - If template source: show template name and configured variables
  - Visual: bordered card with slight shadow, receipt-like vertical layout, subtle divider lines between sections
- REQ-3: **"View recipe" collapsible section** below the summary card. Collapsed by default. When expanded, shows:
  - The generated `.mozza` recipe source in a code block (monospace, dark background)
  - An "Edit" button that makes the recipe editable inline (textarea)
  - A "Copy" button
  - If user edits the recipe, show a "Modified" badge and a "Reset" button to revert to generated version
- REQ-4: Recipe is generated by calling existing `generateOrderRecipe()` (or the equivalent recipe generation function from the current OrderWizard) with the current wizard state. This happens when Step 3 mounts or when coming back from edits.
- REQ-5: **Navigation buttons** at the bottom:
  - Left: "← Back" ghost button (returns to Step 2)
  - Right: **"Place Order"** button — orange (#ff6b35), large, prominent, with subtle glow shadow. Text: "Place Order" (no emoji in the button itself unless the user's design preference includes it)
- REQ-6: "Place Order" calls `onComplete(appName, recipeSource)` — the same callback that DeployWizard.tsx expects. This triggers the deploy pipeline.
- REQ-7: The 3-step indicator shows Step 3 as active, Steps 1 and 2 as completed (checkmarks).
- REQ-8: If recipe has validation errors (from `validateRecipe` API if available), disable "Place Order" and show inline error message.

## Explicit Behaviors

- The order summary is generated from wizard state, NOT from parsing the recipe. It reads `selectedCombo`, `service.isPublic`, `service.dependencies`, target, etc. directly.
- The recipe view is independent of the summary — editing the recipe does NOT update the summary card (and vice versa). The recipe is the source of truth for what gets deployed.
- If user edits the recipe and clicks "Place Order", the EDITED recipe is what gets passed to `onComplete`, not the generated one.
- Going back to Step 2 and changing settings regenerates the recipe (overwriting manual edits, if any). A confirmation could be shown but is not required.
- "Place Order" should be disabled while recipe validation is in-flight (if validation API is called).

```

```

## Constraints

- Summary card must be readable by a non-engineer — no K8s jargon
- Recipe section must be collapsed by default — non-engineers shouldn't see it
- "Place Order" must be the most prominent UI element on the page

## Edge Cases

- Single-service simple deploy → summary is very short (3-4 lines). Still looks good.
- Multi-service deploy (from template) → summary lists each service
- User edits recipe to be invalid → "Place Order" disabled, error shown below recipe editor
- User edits recipe, goes back to Step 2, comes forward → recipe regenerated (edit lost)
- Empty recipe source (shouldn't happen but guard) → "Place Order" disabled
